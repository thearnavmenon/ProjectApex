# Onboarding & Activation Audit — first-session lens

**Scope note:** This reviews the *shipped* code (the P4-T09 flow in `OnboardingView.swift`). The repo also contains a new onboarding design spec written today (`docs/design/onboarding-calibration.md`, status "designed 2026-06-10", explicitly not yet built — `docs/design/ui-overhaul-spec.md:8-9` says "no slices or issues exist for it yet"). Where the spec already answers a finding I say so; several findings below are things the spec does *not* cover and will survive the rebuild if not fixed.

## 1. The first-run funnel as built (step-by-step, from code)

1. **Launch → onboarding gate.** `ContentView` reads a UserDefaults flag once (`ContentView.swift:44`) and presents `OnboardingView` as a non-dismissable full-screen cover (`ContentView.swift:122`, `OnboardingView.swift:113`). No account creation anywhere — an anonymous UUID is minted later (`OnboardingView.swift:909-912`). Genuinely good.
2. **Step 1 — Name** (`OnboardingView.swift:165-212`). "What's your name?" is the only hard-required field in the entire flow — Continue is disabled until non-empty (`:206`). User gets back: nothing.
3. **Step 2 — Training profile** (`OnboardingView.swift:216-430`). Six inputs on one scroll: experience picker, goal picker, days/week (3/4/5/6), bodyweight (optional, kg/lbs), height (cm), age. Promise on screen: "This helps the AI prescribe the right program intensity from day one" (`:223`). User gets back: nothing.
4. **Step 3 — Notification permission** (`OnboardingView.swift:434-483`). Primed screen ("rest timer alerts, session reminders"), real iOS prompt on accept (`:842-850`), soft "Not Now". User gets back: nothing.
5. **Step 4 — Gym scan** (`OnboardingView.swift:487-600` → `ScannerView.swift`). Camera permission, then one photo per piece of equipment, each photo doing a Claude-vision round trip with a spinner (`ScannerView.swift:265-284`), a review card per item, then a confirm-list. Skippable with a warning alert (`OnboardingView.swift:133-141`). This is minutes of physical labor *in a gym* — at first run, the user is statistically on their couch.
6. **Step 5 — Program generation** (`OnboardingView.swift:604-676`, `852-893`). Persists user + goal, then one LLM call (claude-sonnet-4-5, 16k tokens, **180s timeout** — `ProgramGenerationService.swift:823-831`) behind a static spinner saying "usually takes under a minute" (`:635`). **If the scan was skipped, generation is silently skipped** (`:864-869`).
7. **Step 6 — "You're Ready to Train"** (`OnboardingView.swift:680-734`). Says "Your 12-week program is loaded" (`:697`) *unconditionally* — including when no program exists.
8. **Program tab.** What was generated is a *skeleton*: 12 weeks of day-labels, every day `.pending` with **zero exercises** (`MacroPlanService.swift:325-332`).
9. **First workout, intended path:** Program tab → day detail → "Generate Session" (`ProgramDayDetailView.swift:761-767`) → second LLM wait, static "Preparing Your Session" screen (`:357-384`), **120s timeout** (`AppDependencies.swift:173`) — standing on the gym floor → exercises appear → "Start Workout".
10. **In-session:** every set's weight prescription is its own AI call (8s timeout, `AppDependencies.swift:124-127`); preflight spinner (`PreWorkoutView.swift:169-181`); failures surface a Retry sheet, never a silent fallback (`WorkoutSessionManager.swift:1151-1170`). The first AI-prescribed working set is the first moment of unique value.
11. **Calibration framing:** one amber banner — "First session / We'll calibrate your starting weights today" (`PreWorkoutView.swift:316-341`). The calibration *review* (targets) arrives only after ≥4 of 6 patterns reach established confidence, i.e. several sessions in (`CONTEXT.md:38`, `CalibrationReviewBannerCopy.swift:20`).

**Time-to-first-value, happy path:** ~2-4 min of forms + 5-15 min of scanning + ~1-3 min skeleton wait gets you a calendar of day-labels (weak value). The first *strong* value — exercises and weights chosen for your equipment — requires being at the gym, a second 1-2 min AI wait, then an 8s-per-set AI loop. Realistically **two separate contexts (home + gym) and three distinct AI waits before any payoff**, and zero value before a full gym session.

## 2. Findings

**F1 — P0. A fresh install cannot complete onboarding at all: the AI keys live behind onboarding.**
Evidence: every AI call is keyed from Keychain — `(try? keychain.retrieve(.anthropicAPIKey)) ?? ""` (`AppDependencies.swift:106`), and the same for the Supabase anon key (`:96`); keys are entered in Settings → Developer (`SettingsView.swift:9`), reachable only *after* onboarding dismisses. The gym-scan Vision call is also Anthropic-keyed (`VisionAPIService.swift:73-75, 98-100`). With an empty key, step 4's photo analysis fails and step 5 dies into "Generation Failed" showing a raw `error.localizedDescription` (`OnboardingView.swift:891`), whose "Try Again" can never succeed. Known alpha reality, but it means the funnel as designed has no provisioning step and no graceful "this build needs setup" message — anyone outside the dev loop hits a wall with an HTTP error as the explanation. Recommendation: detect missing keys before step 5 (or at app start) and route to a setup screen, rather than letting the user burn 10 minutes of scanning first.

**F2 — P0. The skip-scan path ends in a three-screen contradiction, with copy that lies.**
Evidence: skip → generation silently skipped (`OnboardingView.swift:864-869`) → step 6 still announces "Your 12-week program is loaded. Head to the Program tab" (`:697`) → Workout tab says "Generate a program in the Program tab" (`ContentView.swift:409-413`) → Program tab empty state says "Scan your gym in the Scanner tab to get started" (`ProgramOverviewView.swift:175`) — **there is no Scanner tab**; that's Phase-1 copy (the old tab layout survives in `ARCHITECTURE.md:155`), the scanner now lives in Settings behind "Complete your setup" (`SettingsView.swift:142-165`). The user who took the only escape hatch you offered gets told something false, then sent on a scavenger hunt across three surfaces with three different instructions. Skip is the *normal* path for an at-home first run, so this is the modal new-user experience. Recommendation: make step 6 truthful per-branch, and give the empty Program state one working CTA (open scanner / open manual picker directly).

**F3 — P0. The Workout tab is a first-session trap that also corrupts next launch.**
Evidence: after onboarding, every day is `.pending` with no exercises; `nextIncompleteDay` happily returns it (`ProgramViewModel.swift:732-747`), so the tab literally named "Workout" shows PreWorkoutView with "0 exercises" and "~0 min estimated" (`PreWorkoutView.swift:261, 300, 544-551`) plus the first-session calibration banner. Tapping Start calls `startSession`, which **writes the crash sentinel and enqueues a Supabase session row** (`WorkoutSessionManager.swift:244-261`) *before* the empty-exercises guard errors out (`:263-266`) — and the error path never clears the sentinel. Next launch: "Unfinished Workout — Resume or Abandon?" (`ContentView.swift:181-200`) for a workout that never happened. A brand-new user's most likely tap produces an error screen and then a recovery dialog about a phantom session. Recommendation: in PreWorkoutView, a pending day's CTA should *be* "Generate Session" (reuse the day-detail path); and move the sentinel write below the guard.

**F4 — P0. Onboarding answers are partially discarded by the system that builds the actual first workout.**
Evidence: `generateDaySession` — the call that picks the user's first exercises and weights — reads UserDefaults keys that **nothing in the codebase writes** (`"bodyweight_kg"`, `"user_age"`, `"training_age"` at `ProgramViewModel.swift:424-428`; onboarding writes `com.projectapex.user.*` via `UserProfileConstants`, `OnboardingView.swift:918-921`), so biometrics are always nil there; and it hardcodes `experienceLevel: "intermediate"`, `goals: ["hypertrophy"]` (`:430-437`). Post-onboarding regeneration paths also hardcode the goal (`:290`, `:350`). So the beginner who chose "Strength" is, at session-generation time, an intermediate hypertrophy trainee. The goal only influences the onboarding-time skeleton call (`OnboardingView.swift:878`) and a trainee-model write. This breaks the on-screen promise at step 2 (`:223`) and quietly undermines "every question shows its payoff" — the founding principle of the new spec — *regardless* of which onboarding UI sits in front of it. Recommendation: fix the key mismatch, persist the goal, and thread real profile values through every generation path; this is a bug fix, not a redesign.

**F5 — P1. Cold start: the coach prescribes weights for a stranger, and the only framing is one sentence.**
Evidence: no screen asks anything about current lifts; first-session loads derive from demographics (and per F4, often not even those). The only expectation-setting is the banner "We'll calibrate your starting weights today" (`PreWorkoutView.swift:316-341`), which disappears after sessionCount > 0 (`WorkoutView.swift:65-67`) — even after a botched 2-set session. There is no degraded "week one" experience for a user who skips or flubs calibration; the model just ingests whatever happened. The design spec names this exactly (review finding P0-3, "cold-start cliff") and fixes it with confidence-tagged capability seeding + a model-reveal payoff + honest day-one copy ("First session doubles as calibration — expect a weight check or two", `onboarding-calibration.md:128-131`). The gap is that **nothing of it is scheduled**: spec'd today, zero slices. The finding is the doc/code gap itself — until it ships, day one rests on a single banner and the in-session retry/correction machinery.

**F6 — P1. The equipment ask is maximal, mistimed, and has no manual fallback on a real device.**
Evidence: the device idle screen offers only "Start Scanning" (`ScannerView.swift:92-126`); manual entry exists only on the simulator path (`:129-164`, `skipToManualEntry` `ScannerViewModel.swift:156-158`) or *after* at least one successful photo reaches the confirm list (`ScannerView.swift:393-397`); the permission-denied screen's "Enter Equipment Manually" button actually calls `reset()` → back to the camera idle screen (`ScannerView.swift:514-517` → `ScannerViewModel.swift:230-235`) — the label is a lie and the user loops. So an at-home first-run user's real options are "photograph a gym you're not in" or "skip" (→ F2). The cruel part: a perfectly good manual picker already exists (`BulkEquipmentPickerSheet.swift`, used in Settings and in the confirm list). The spec's answer — a 6-item multi-select grid with a "full commercial gym" shortcut (`onboarding-calibration.md` §3.3) — is right; the camera scan should be a post-activation enhancement, never the gate.

**F7 — P1. Quitting onboarding loses everything; re-entry restarts at step 1.**
Evidence: all flow state is view-local `@State` (`OnboardingView.swift:66-76`); the completion flag is written only at the very end (`:964`). Force-quit at step 5 and you redo name, profile, permission — and the gym scan, because although the scanned profile *was* saved to UserDefaults at confirm time (`ScannerViewModel.swift:274`), onboarding never reads it back; step 4 re-presents "Open Gym Scanner" with `gymProfile = nil` (`:68`). A user who photographed 15 machines and got a phone call must re-scan or skip. Recommendation: persist step + profile incrementally, and seed step 4 from `GymProfile.loadFromUserDefaults()`.

**F8 — P1. The waits are static, the estimates contradict the timeouts, and nothing is recoverable.**
Evidence: skeleton wait says "usually takes under a minute" against a 180s timeout (`OnboardingView.swift:635` vs `ProgramGenerationService.swift:828`); the gym-floor session-generation wait is a static three-line screen against a 120s timeout (`ProgramDayDetailView.swift:357-384` vs `AppDependencies.swift:173`); neither narrates progress, survives a backgrounding/kill, or notifies on completion. The user's longest waits are also their lowest-trust moments. The spec's "drawing your model" checklist ("real derived values… no fake percent counters", `onboarding-calibration.md` §3.7) covers the reveal, but says nothing about the *session-generation* wait at the gym — that one needs its own treatment (pre-generate day 1 during onboarding's step 5, or narrate beats: "reading your history → placing weights → ordering exercises"). Pre-generating the first session would also delete an entire gym-floor wait.

**F9 — P2. Notification permission is burned at the lowest-trust moment.**
Evidence: step 3 of 6, before the user has received anything (`OnboardingView.swift:434-483`). The priming screen and soft "Not Now" are good practice, but the stated value ("rest timer alerts") first becomes real mid-workout — the textbook contextual-ask moment. The new spec's flow drops notifications from onboarding entirely (its 8 screens contain no permission ask); make sure the rebuild actually moves the prompt to first rest-timer start rather than silently dropping it.

**F10 — P2. The "ready" screen's copy is wrong in both unhappy branches.**
Evidence: "Your 12-week program is loaded" (`OnboardingView.swift:697`) renders after skip-scan (generation never ran, `:864-869`) and after "Continue without a program" on generation failure (`:657-665`). The scanSkipped warning chip below (`:704-722`) contradicts the headline two lines above it. One-line fix per branch; cheap trust repair.

**F11 — P2. Step 2 collects data with no consumer and the wrong option set.**
Evidence: height and age are asked (`OnboardingView.swift:341-413`) but the design spec states "height, age, sex are not asked in v1 — add only when an engine consumer exists" (`onboarding-calibration.md` §4) — and per F4, age never even reaches session generation. Days/week offers 3/4/5/6 (`:248`) vs the spec's 2/3/4/5+ — excluding the 2-day option that fits the most fragile new lifters. Six inputs on one screen vs the spec's one-question-per-screen. Defer height/age; align the schedule options when the rebuild lands.

**F12 — P2. The only required field is the least valuable one, asked first.**
Evidence: display name gates Continue on screen one (`OnboardingView.swift:206`), before the app has shown anything; goal — the answer that actually steers the product — is buried mid-step-2 and never re-consumed (F4). The spec opens with goal and asks for no name at all (`onboarding-calibration.md` §3.1 and the Centr note "we don't have a name yet"). Counterweight worth keeping: no account, no email, no paywall anywhere in this flow — that's a real activation asset the rebuild should preserve (spec §4 agrees: no auth before the payoff).

## 3. The three things to fix first

1. **Make skipping the scan a working path instead of a lie.** Today the most likely first-run choice (user at home, skips scan) gets false copy ("program is loaded"), a Workout tab that errors and plants a phantom "unfinished workout", and an empty state pointing at a tab that doesn't exist. Fix the step-6 copy per branch, put the existing `BulkEquipmentPickerSheet` into onboarding step 4 as an equal-weight "I'll pick my equipment" option, guard the Workout tab's pending-day Start (and move the crash-sentinel write below the empty-exercises check). These are small diffs that remove every dead end from the modal path — worth doing now, before the redesign.

2. **Stop throwing away the answers.** Fix the UserDefaults key mismatch and the hardcoded "intermediate / hypertrophy" in `generateDaySession`/`generateMacroSkeleton`/`generateProgram`, and persist the goal. Until what's asked actually changes what's prescribed, no onboarding — old or new — can keep its core promise, and the new spec's "every screen must visibly buy intelligence" principle is unbuildable on top of this.

3. **Collapse the gym-floor wait.** The user's first minutes physically in the gym are spent watching a static spinner for up to two minutes. Pre-generate day 1's session during onboarding's existing generation wait (one wait instead of two, in the comfortable context instead of the gym), and give whatever wait remains the spec's honest-checklist narration. Then schedule the `onboarding-calibration.md` build — it correctly fixes the deepest problem (cold start, F5/F6), and it's currently a spec with zero slices behind it.
