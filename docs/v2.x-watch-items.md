# v2.x watch-items

Items deferred from Phase 2 design with explicit revisit triggers. Each carries a one-sentence description and a cross-reference to the originating ADR or PRD-internal decision. Catalogued here to prevent loss between v2 shipping and v2.x design starting.

These are not bugs and not roadmap items. They are *known limitations or deferred refinements* with a defined surface in the trainee-model rule logic. v2.x design starts by walking this list and deciding which items have surfaced as real problems in the alpha cohort.

## Recovery and stimulus

### 1. Stacking detection via inter-session-gap residuals
Recovery readiness is volume-blind and stacking-blind: two NM-classified sessions 18h apart both ratchet `lastNeuromuscularStimulusAt` forward without compounding residual debt. The monitoring trigger is prescription-accuracy residuals stratified by inter-session-gap bucket (per ADR-0014's gap-bucket schema). If `<48h` bucket bias systematically diverges from `>72h` bucket bias on the same pattern, that's stacking error and v2.5 priority — full implementation requires a `*ReadinessDebt: Double` field on `RecoveryProfile` and a Codable migration. **Source: ADR-0010 §Stacking — deferred to v2.5 with explicit monitoring.**

### 2. Volume-modulated recovery tau
A user doing 3 sets of bench at RPE 8 and a user doing 12 sets at RPE 8 follow the same NM recovery curve under v2's model. Real residual fatigue scales with volume. Likely v2.x fix: `tau_NM = base_tau × (1 + 0.05 × heavy_set_count_above_threshold)` rather than additional state. Revisit if alpha cohort shows systematic under-recovery on high-volume programmes. **Source: ADR-0010 §Volume-blindness — acknowledged limitation.**

## Prescription accuracy

### 3. RPE-error signal alongside rep-error
Rep-error misses the under-effort case: a user delivering 6/6/6/6 reps at RPE 7 looks identical to a user at RPE 9, but the AI under-prescribed in difficulty terms for the first user. When RPE logging coverage in alpha is reliable enough to build on, an `rpeError: Double` field alongside `bias`/`rmse` on `PrescriptionAccuracy` would catch this. v2 ships rep-only because RPE coverage is currently inconsistent. **Source: ADR-0014 §Error metric, known limitation.**

### 4. Per-exercise prescription accuracy granularity
Bias is aggregated at `(pattern × intent)`, not `(exercise × intent)`. A user well-calibrated on barbell bench but systematically over-delivering on dumbbell incline press would have those signals mixed within `horizontalPush.top` and the mean bias might come out near zero. Multiplies cell count and slows cold-start; deferred. Revisit if alpha cohort shows the meta-coaching feedback isn't biting on systematic exercise-specific miscalibration. **Source: ADR-0014 §Considered Options + §Consequences future surface.**

### 5. Peaking-phase noise (known caveat, not deferred work)
Peaking-phase prescription-accuracy bias is structurally noisier than accumulation-phase bias because peaking deliberately pushes toward the lifter's ceiling. v2 includes peaking sets in the bias estimate (rejected the alternative of excluding them) on the grounds that prescription should still track near-maximal capability. Future tuning should expect higher noise on peaking-heavy data and avoid tightening surfacing thresholds based on it. Not a deferred refinement — documented as a known characteristic. **Source: ADR-0014 §Peaking-phase caveat.**

## Classifier and limitations

### 6. Classifier sustained-failure meta-coaching surface
v2 keeps classifier failures invisible to the user (background-tier per ADR-0013, RAG covers the acute case). If sustained failure (e.g., `lastClassifiedNoteCreatedAt` hasn't advanced in N sessions) occurs in alpha, the user is genuinely accumulating un-aggregated notes and the meta-coaching layer might surface "your form patterns aren't being analysed lately." v2 relies on the structured `trainee_model.classifier_failed` log channel + manual monitoring during alpha; v2.x adds the user-facing surface if the log shows the failure mode is real. **Source: ADR-0013 §Why background-tier, sustained-failure deferral.**

### 7. Lower-back-as-muscle taxonomy gap
The current `MuscleGroup` enum is locked-six (back, chest, biceps, shoulders, triceps, legs) — no `lowerBack`. Under Q9's language-driven classifier scoping rule, "lower back sore" (tissue-language) has no muscle subject to land on and falls through to `.joint(.lowerBack)`. Joint-scope is conservative-correct under asymmetric-error reasoning, but the type system can't currently express tissue-scoped lower-back limitations. v2.x may add `MuscleGroup.lowerBack` if alpha cohort surfaces this gap. **Source: Q9 PRD-internal, classifier-prompt code comment.**

## Phase advance

### 8. Consecutive force-deload automated remediation
ADR-0011 introduces `consecutiveForceDeloadsOnPattern: Int` and surfaces it to the LLM digest at counter ≥ 2 for natural-language exercise-rotation/programme-rebuild coaching. v2 leaves the actual remediation (exercise rotation, programme rebuild) to the AI's natural-language coaching rather than automating it. Future versions may add deterministic rotation logic — e.g., automatic exercise substitution from a pool of equivalents — but v2 ships the signal only. **Source: ADR-0011 §(d) Watch-item: consecutive force-deloads.**

## Muscle-level aggregation

### 9. Muscle stagnationStatus aggregation under worst-across-patterns

ADR-0009's amendment (2026-05-07) pins `MuscleProfile.stagnationStatus` aggregation as worst-across-patterns (`declining > plateaued > progressing`) over participating patterns whose confidence > `.bootstrapping`. The "one bad pattern poisons the muscle signal" risk concentrates on muscle groups with multiple participating patterns: `legs` (3: squat, hipHinge, lunge) and `back` (2: horizontalPull, verticalPull). Squat plateaus are particularly common; one squat plateau forces `legs.stagnationStatus = .plateaued` even when hipHinge and lunge are progressing. **Trigger:** if leg-muscle `stagnationStatus` reads `.plateaued` >30% of the time across the alpha cohort's session-applies, re-apply ADR-0009's hybrid verdict at muscle level (compute muscle-level e1RM track + already-per-muscle volume-load track, run the OR/AND aggregation table at muscle level). Sub-decision required at upgrade time: muscle-level e1RM aggregation rule (max-of-exercises vs mean vs primary-exercise dominance). **Source: ADR-0009 §"Aggregation to MuscleProfile.stagnationStatus", v2.x upgrade path.**

## Late-arrival sessions

### 10. Backfill mode for refused late-arrival sessions
ADR-0008 refuses session-completion events with `loggedAt < watermark` and surfaces a soft notification to the user. The historical `set_logs` rows persist; the model state misses. v2.5 may add a backfill mode that re-derives the model from scratch when refused events accumulate above some threshold — the schema choice (`last_applied_logged_at` watermark + dedupe table) supports this without further migration. Revisit if alpha cohort reports cluster around late-arrival scenarios (e.g., extended connectivity outages). **Source: ADR-0008 §Consequences, v2.5 backfill hook.**

## How to use this list

When v2.x design starts, walk these 10 items in order and ask for each:

1. **Has alpha cohort surfaced this as a real problem?** Check the relevant observability channel (`trainee_model.classifier_failed`, prescription-accuracy gap-bucket divergence, etc.) and any user-reported incidents.
2. **What's the v2.x-shaped fix?** Each item carries a likely-fix sketch in its source ADR; revisit it against current alpha learnings.
3. **What's the migration cost?** Most items are additive (new fields with defaults); a few require Codable migrations or rule-recompute paths.

Items that haven't surfaced as real problems by v2.x design time stay deferred. The list shrinks as items either ship or get formally re-deferred to v3.
